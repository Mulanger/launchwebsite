# Polywhale Website

Web app for Polywhale. Production now runs the Next.js SSR/SSG foundation on Railway so public routes return meaningful HTML before hydration. The older React/Vite build remains available for local fallback and comparison.

The site consumes the Railway whale-trade API and presents a public live feed, leaderboard, trader profiles, trade details, web alerts setup, following management, and legal pages.

## Local development

Legacy Vite app:

```bash
npm install
npm run dev
```

The dev server proxies `/api/*` to the production API by default. To point it at another API:

```bash
$env:VITE_DEV_API_TARGET="http://127.0.0.1:3000"; npm run dev
```

Current Next app:

```bash
npm run next:dev
npm run next:build
npm run next:start
```

The Next app server-renders/static-renders public SEO routes and hydrates app-only routes through the existing React app.

Market landing pages:

```text
/market/[slug]
```

Market pages are native Next server routes. Known markets render the same Polywhale dashboard frame with market-specific whale stats, recent whale trades, top market wallets, and a market snapshot rail. Unknown markets return 404. Weak known markets stay `noindex,follow`; qualified markets require at least 3 tracked whale trades and at least $50K tracked whale volume.

The market page data path now prefers the enriched API endpoint `/v1/market-pages/:slug`, populated by the whale-watcher `market_page_snapshots` worker. If that endpoint is unavailable during deployment rollout, the website falls back to the previous whale-feed scan.

## Production build and deployment

Current production build path:

```bash
npm run next:build
npm run next:start
```

Railway can deploy this repository with the included `railway.json`.
The production Next server proxies `/api/*` through `app/api/[...path]/route.js` to `https://whaleserver-production.up.railway.app` unless `API_BASE_URL` is set.

Railway currently uses:

```text
Build: npm run next:build
Start: npm run next:start -- -H 0.0.0.0 -p $PORT
```

## SSR migration status

The migration plan is documented in `NEXT_SSR_MIGRATION_PLAN.md`.

Implemented and deployed from `codex/next-ssr-foundation`:

- Next.js installed alongside the existing Vite app.
- Public SEO routes exist in `app/`: `/`, `/about`, `/leaderboard`, `/privacy`, `/terms`, `/delete-data`.
- Native market SEO routes exist at `/market/[slug]` and use existing whale data for the MVP.
- App-only routes are wired in Next through a client wrapper: `/alerts`, `/profile`, `/profile/following`, `/trade/[tradeId]`, `/trader/[wallet]`.
- `/` and `/leaderboard` use a hybrid approach: crawlers receive server-rendered HTML, while browser users hydrate into the existing app UI.
- `/api/*` is preserved through a Next API proxy route.
- `/sitemap.xml` is now generated by Next and includes only qualified market pages.
- Market pages and sitemap prefer enriched `market_page_snapshots` via `/v1/market-pages`, with scan fallback for safe deploy ordering.
- Railway has been cut over from Vite commands to Next commands.

Validated:

- `npm run next:build`
- `npm run build`
- Local `next start` route checks on `http://localhost:3000`
- Direct route refreshes for public and app-only routes
- `/api/*` proxy response
- Production websocket endpoint opens
- Production source HTML contains `next-app-snapshot` and `/_next/static` for `/` and `/leaderboard`
- Production source HTML no longer contains the old Vite `<div id="root"></div>` shell

## Before publishing

- Replace launch/update links when the Google Play listing is live.
- Update `support@whaletracker.com` if the public support email changes.
- Review `Privacy Policy`, `Terms`, and `Delete Data` pages before submitting them in Play Console.
- Use the live `/privacy` URL as the app privacy policy URL in Play Console.
- Use the live `/delete-data` URL if Play asks for a data deletion URL.
- After SEO or framework changes, verify production source HTML, API, websocket, follows, alerts, and mobile views.
